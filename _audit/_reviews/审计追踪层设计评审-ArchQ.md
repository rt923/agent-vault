# 审计追踪层（vault/）设计评审 — ArchQ

> 评审对象：用户贴出的 `vault/` 审计追踪层目录结构设计稿
> 评审视角：架构治理（分层 / 数据流向 / 幂等 / 可追踪性 / 可维护性）
> 数据来源：设计稿结构本身（非 CloudQ 云资源扫描；CloudQ 评估的是线上云资源，本 vault 是本地文件型治理系统）
> 评审时间：2026-07-20

---

## 一、总体评价

| 维度 | 评级 | 说明 |
|------|------|------|
| 分层合理性 | ✅ 良好 | 源数据 → 同步核心 → 派生看板，职责分离清晰 |
| 幂等意识 | ✅ 良好 | 同步脚本明确标注幂等，是治理系统基本功 |
| 可追踪性 | ⚠️ 有缺口 | 提案 ↔ 闸口事件 之间缺双向引用，审计链未闭环 |
| 数据契约 | 🔴 缺失 | 同步的 .md 无强制 schema，看板聚合脆弱 |
| 跨平台可移植性 | ⚠️ 需注意 | `.sh` 在 Windows 上需 Git Bash / 重写成 Node·Python |

**一句话**：骨架方向对，但要在"数据契约"和"追踪闭环"两处补强，否则看板算不准、审计追不回。

---

## 二、设计亮点（符合架构治理原则）

1. **派生数据不当真相源** — `_dashboards/` 是"自动计算"出来的，不是手存的。避免派生数据变成第二真相源，这是正确数据架构。
2. **幂等同步** — `sync-audit.sh` 标注幂等，可重复跑不污染，符合管线基本要求。
3. **按月分片** — `_gate-events/2026-07/` 避免单目录无限膨胀，利于规模扩展。
4. **内容寻址命名** — `gate-...PASS-20bceea9.md`（结果+短哈希），天然支持去重与幂等跳过。
5. **每层带 `_index.md`** — 可导航性到位。

---

## 三、风险与反模式（按优先级）

### 🔴 P0 — 正确性与完整性硬伤

**P0-1 缺强制数据契约（schema）**
- 风险：看板 `gate-trends.md`（通过率、拦截分布）要从 `gate-*.md` 聚合，但文件内部格式未定义。无 frontmatter，脚本只能靠文件名正则猜，一旦命名漂移看板就失真。
- 建议：规定每个同步文件必须带 YAML frontmatter，至少含：
  ```yaml
  ---
  outcome: PASS | REJECT
  ts_event: 2026-07-13T14:02:00+08:00   # 事件发生时间
  ts_sync:  2026-07-20T18:33:00+08:00   # 同步时间
  rule_id: guardrails-readonly-windows
  actor:   agent | human
  proposal_id: PEG-2026-07-13-001        # 若是自指提案的闸口结果
  source_hash: 20bceea9
  ---
  ```

**P0-2 源真相源策略未定义**
- 风险：树标注"_gate-events（从 logs/ 同步）"，暗示单向 logs→_audit。但若有人在 `_audit/` 里手改文件，下次同步会覆盖或跳过，导致"审计记录被悄悄改掉"。
- 建议：明文规定 `_audit/` 是**只读镜像**；任何修正必须回到源（logs/drafts/fix_reports）改，再重同步。同步脚本对 `_audit/` 已有文件默认"跳过不改"，除非源显式更新。

**P0-3 分片键用错（同步时间 vs 事件时间）**
- 风险：`2026-07/` 若是"同步发生的月份"，则一条 6 月的日志在 7 月同步会被塞进 `2026-07/`，月份语义错乱，月度看板失真。
- 建议：按 **`ts_event` 的月份**分片，而非同步时刻。脚本里用事件时间算目录。

### 🟡 P1 — 治理完整性缺口

**P1-1 提案 ↔ 闸口事件 无双向引用（审计链断裂）**
- 风险：`_proposals/self-modify-001.md`（自指提案）本该由某个 gate-event 审批/驳回，但两边没有互指 ID。你无法回答"这条自改提案是被哪次闸口放行的"。
- 建议：提案 frontmatter 加 `gate_event_id`；闸口事件 frontmatter 加 `proposal_id`（见 P0-1）。闭环。

**P1-2 看板重算触发未定义**
- 风险：同步跑完但看板重算失败/未跑 → 看板与 `_audit/` 漂移。
- 建议：`sync-audit.sh` 末尾原子重算 `_dashboards/`，并在看板头部写 `generated_at:` 时间戳；或明确看板是"按需生成"并标注陈旧风险。

**P1-3 删除/裁剪策略缺失**
- 风险：`_index.md` 如何随源删除更新？审计数据通常**只增不删**。若同步只"追加不 prune"，源删了索引还在，造成幽灵条目。
- 建议：明确 `_audit/` 为**追加只读 + 显式 tombstone**，不在同步中自动删历史事件。

### 🟢 P2 — 运维与可移植性

**P2-1 `.sh` 在 Windows 的可移植性**
- 风险：环境为 Windows。`.sh` 需 Git Bash/WSL 才能跑；且中文路径 + GBK 易乱码（你的项目约定是 Node fs utf8 禁 PowerShell）。
- 建议：用 Git Bash 跑（本机可用），或把同步脚本重写成 Node/Python，统一 UTF-8，契合你现有工程约定。

**P2-2 无保留/归档策略**
- 风险：`_gate-events/` 永久增长，旧月无冷存/上限。
- 建议：定义保留期（如 13 个月热存 + 归档），或按月压缩。

**P2-3 同步脚本缺 dry-run**
- 风险：直接跑难预判影响。
- 建议：加 `--dry-run` 只打印将同步/跳过项。

## 四、需你确认的决策点

1. **源目录在哪？** 同步要从 `logs/` `drafts/` `fix_reports/` 拉——这些目录当前不在工作区，是在别处（如 agent 运行目录）吗？路径要写进脚本。
2. **同步方向？** 单向 logs→_audit（推荐），还是双向？
3. **看板生成时机？** 随同步原子生成，还是独立定时任务？
4. **运行环境？** Git Bash 跑 `.sh`，还是我帮你重写成 Node/Python（更稳、UTF-8 安全）？
